**Goal:** Close A3 (CE-P2 list/vectorization semantics) from the CE 0.58.0 plan. Six sub-items:
1. `ce.maxCollectionSize` config (default 10,000) with typed-Error result at materialization sites
2. Empty-list validity — `\left[\right]` is a valid value

6. Function-application broadcasting — `f(L)` evaluates by mapping `f` over `L` when the parameter type is scalar

**Architecture:** All work is source-side (no compile-target changes for A3).

- **`ce.maxCollectionSize`**: new field on `EngineRuntimeState`, exposed via getter/setter on `ComputeEngine`. Guards added at materialization sites (`Range` collection conversions, `Repeat`, eager `materialize()` paths, any List-constructing reducer).
- **Empty list**: existing iterator handling already supports `List()`. Add regression tests; tighten the type if widening produces `list<unknown>` instead of `list<never>` for the empty case.
- **Reducer arity-1**: `Sum`'s current signature is the big-op form `Sum(f, bounds)`. Add a single-operand branch in `Sum`'s canonical/evaluate that recognizes a collection argument and reduces it. `Max`/`Min` already accept collections through `processMinMaxItem`; verify single-collection-argument form works. `Count(L)` already exists.
- **Type refinement**: per the answered design question, no new API. Verify that `widen(...elements)` produces a clear union for mixed-kind lists and that mixed-dim tuples surface in the element type. Add tests asserting GP can detect via `.type`.
- **Indexing**: extend `At`'s evaluate handler so that when `index` is a finite collection of integers (multi-pick) or finite collection of booleans (mask), it returns a filtered/picked list. Numeric index keeps existing semantics.
- **Function broadcasting**: in `applyFunctionLiteral`, detect when any operand is a finite indexed collection and the function's parameter type at that position is scalar (not `list`/`collection`/`tuple`). When true, zip the arguments (broadcasting scalars) and map. Falls back to the existing path when params are list-typed.

## Task A3.1: `ce.maxCollectionSize` config + materialization guards

GP needs a structured "too many points" diagnostic instead of OOM. Audit-derived default: **10,000** (corpus max is 4,000; P99 is 225).

**API design:**

```typescript
interface IComputeEngine {
  /**
   * Maximum number of elements a collection may have when materialized
   * (converted from a lazy form to a `List`). Defaults to 10,000.
   *
   * When a materialization would exceed this size, the operation that
   * triggered it returns `undefined` (leaving the lazy form intact) and
   * may emit a typed `Error` form, so consumers can detect oversize
   * collections via `.operator === 'Error'` or by inspecting the held
   * symbolic form's count.
   *
   * Set to `Infinity` or `0` to disable the cap (not recommended for
   * untrusted input).
   */
  maxCollectionSize: number;
}
```

**Guard sites:**
- `Repeat`'s `evaluate` (replaces the existing 10k constant)
- `Range`'s implicit materialization (any callsite that turns a Range iterator into a List)
- `materialize()` in `boxed-function.ts` when it's about to fully materialize a finite indexed collection
- Any reducer that constructs a List from an iterator (e.g., `Map` evaluate to a list)

**Files:**
- Modify: `src/compute-engine/engine-runtime-state.ts`
- Modify: `src/compute-engine/index.ts`
- Modify: `src/compute-engine/types-engine.ts`
- Modify: `src/compute-engine/library/collections.ts` (Repeat + any Range materialization paths)
- Modify: `src/compute-engine/boxed-expression/boxed-function.ts` (materialize)
- Test: `test/compute-engine/a3-lists.test.ts` (new file)

- [ ] **Step 3: Add arity-1 collection branch to `Sum`**

In `src/compute-engine/library/arithmetic.ts` around line 1946–1997, modify the `Sum` operator. The simplest pattern: detect at canonical (or in evaluate, if canonicalization is tricky) when `ops.length === 1` AND `ops[0]` is a collection. Reduce by iterating elements.

Implementation approach — add the arity-1 branch in `evaluate`:

```typescript
      evaluate: ([first, ...rest], { engine, numericApproximation }) => {
        // Arity-1 collection-reducer form: Sum(L) where L is a collection.
        if (rest.length === 0 && first?.isCollection) {
          let acc: Expression = engine.Zero;
          for (const x of first.each()) {
            acc = acc.add(x.evaluate({ numericApproximation }));
          }
          return acc.evaluate({ numericApproximation });
        }

        // Existing big-op form: Sum(body, [i, a, b], …).
        const result = run(
          reduceBigOp(
            first,
            rest,
            (acc: Expression, x) =>
              acc.add(x.evaluate({ numericApproximation })),
            engine.Zero
          ),
          engine._timeRemaining
        );
        if (result === NON_ENUMERABLE_DOMAIN) return undefined;
        return result?.evaluate({ numericApproximation }) ?? engine.NaN;
      },
```

Mirror the change in `evaluateAsync`.

Also relax the canonical handler: when `ops.length === 1` and `ops[0]` would be a collection, skip the `canonicalBigop` call (which expects an indexed body + bounds). The cleanest pattern:

```typescript
      canonical: ([body, ...bounds], { scope, engine: ce }) => {
        // Arity-1 collection-reducer form: don't run canonicalBigop (which
        // expects bounds). Just canonicalize the operand.
        if (bounds.length === 0) {
          const canon = body?.canonical;
          if (canon?.isCollection) return ce._fn('Sum', [canon]);
        }
        return canonicalBigop('Sum', body, bounds, scope);
      },
```

**Verify before finalizing:**

Update to accept both: `((number) -> number, bounds:tuple+) -> number` becomes a union, or just `(any, any*) -> number` if no clean union form exists. Look at how `Sum` is declared in `definitions-arithmetic` LaTeX (if dot-notation `L.total` already parses to `Sum(L)`, that path must accept arity-1 today).

## Task A3.5: Indexing extensions (`At` with boolean mask / index list)

Per the answered design question, extend `At`. Today `At(L, i)` requires `i` to be an integer. Add:
- `At(L, mask)` where `mask` is a finite collection of booleans → returns a list of elements where the mask is True.

function applyFunctionLiteral(
  expr: BoxedFunction,
  def: BoxedValueDefinition,
  options?: Partial<EvaluateOptions>
): Expression {
  const value = def.isConstant
    ? def.value
    : expr.engine._getSymbolValue(expr.operator);

  if (value && !value.type.matches('function')) {
    if (!value.isValid) return expr;
    return expr.engine.typeError('function', value.type, value.toString());
  }

  const ops = expr.ops.map((x) => x.evaluate(options));
  if (!value || value.type.isUnknown)
    return expr.engine.function(expr.operator, ops);

  // Broadcasting: if any operand is a finite indexed collection AND the
  // function's corresponding parameter type is scalar (not a collection),
  // map the function over the zipped operands.
  if (
    ops.some((x) => isFiniteIndexedCollection(x)) &&
    paramsAreScalar(value)
  ) {
    const items = zip(ops);
    if (!items) return expr.engine.Nothing;

    const results: Expression[] = [];
    while (true) {
      const { done, value: zipped } = items.next();
      if (done) break;
      const applied = apply(value, zipped);
      results.push(applied.evaluate(options));
    }
    if (results.length === 0) return expr.engine.Nothing;
    return expr.engine._fn('List', results);
  }

  return apply(value, ops);
}

## Risk register

| Risk | Mitigation |
| --- | --- |
| `Sum` arity-1 dispatch interferes with big-op `Sum(body, [i,a,b])` where `body` is itself a collection-typed expression | Narrow the discriminator: only treat arity-1 as collection-reducer when `rest.length === 0`. Big-op always has `bounds` (rest non-empty). |
| `applyFunctionLiteral` broadcasting breaks user-defined functions that intentionally consume lists | The `paramsAreScalar` predicate is the gate. If the type-system inspector is unclear, start with always-broadcast and report any breakage; narrow accordingly. Mark `paramsAreScalar` as TODO if implementation is non-trivial — the always-broadcast default is a reasonable shipping behavior. |
| `At` boolean-mask detection misclassifies indices like `[1, 0, 1]` (could be either integers or booleans-ish) | The check is on `symbol === 'True'`/`'False'`, not on integer values. `[1, 0, 1]` is not a mask — it's integer indices. |
| `maxCollectionSize` cap breaks existing tests that materialize large finite ranges | The default is 10,000; corpus P99 is 225 and max is 4,000. No real-world test exceeds this. If a test fixture deliberately constructs >10k elements, bump the cap in the test setup. |
| Type widening change for mixed-kind lists breaks pre-existing tests that assert specific `list<X>` types | Tests in this plan are written defensively (regex / generic shape assertion). If pre-existing tests fail, restore the previous widen behavior — the test changes here are only assertions about what GP can detect. |
| `zip()` semantics on empty collection differ from expected `Nothing` vs. `List()` | The existing `def.broadcastable` path returns `Nothing` for empty; our path needs to return `List()`. Check `results.length === 0` and return an empty `List` explicitly. |

## Open question for GP (not blocking)

`Sum(L)` arity-1 form needs a clean signature. The current `((number) -> number, bounds:tuple+) -> number` is big-op-only. After A3, `Sum` accepts either:
- `Sum(body, bounds...)` — big-op
- `Sum(collection)` — reducer

The cleanest type-system expression is overloading. If overloading is not supported, the signature widens to `(any, any*) -> number` and the discrimination is dynamic. GP doesn't need to do anything either way — both forms work; only the strictness of `expr.type` checking is affected.
