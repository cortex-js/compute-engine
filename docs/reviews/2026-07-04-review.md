## A. Higher-order ODE support (new in 0.67.0) — 2 P0, 3 P1

`src/compute-engine/symbolic/differential-equations.ts`,
`differential-equation-utils.ts`, `library/calculus.ts:526-584`.

- **A1 (P0, confirmed). Numeric-root fallback ignores root multiplicity →
  degenerate general solution.** `solveHigherOrderWithNumericRoots`
  (`differential-equations.ts:584-620`) emits one mode per Durand–Kerner root
  with no clustering. `y'''' + 2y'' + y = 0` (roots `±i` double) returns four
  numerically dependent terms with garbage `e^(3.8e-9·x)` factors instead of
  `(c₁+c₂x)cos x + (c₃+c₄x)sin x`; `(r−1)²(r²+1)` loses the `x·eˣ` mode
  entirely (3-dimensional "general solution" for an order-4 ODE). Point-residual
  verification cannot catch this (each term individually near-satisfies the
  ODE) — a Wronskian/dimension check would. Any characteristic polynomial with
  complex roots and degree ≥ 3 routes to this path (`polynomialRoots` returns
  real roots only).
- **A2 (P0, confirmed). Corrupted `Error`-node "solution" for `≥2nd`-order
  derivative inside a `Multiply`.** `collectLinearTerms`
  (`differential-equations.ts:127`) canonicalizes the residual term; canonical
  `Multiply` typechecks `D(D(y(x),x),x)` (type `expression` when `y` is
  undeclared — the default) into `Error("incompatible-type")`; the
  `hasDependentOrDerivative` guard (1131-1144) then inspects the corrupted tree
  and lets the first-order solver run. `x²y'' + xy' = x` →
  `[y(x) = c₁ + (−Error(…)·x²/2) + x]` returned as a successful `List`.
- **A3 (P1, confirmed).** `y'(x) = y(2x)`: `isDependentFunction` matches only
  literal `y(x)`, so `y(2x)` is treated as external forcing and the first-order
  branch (1150-1160) returns `c₁ + ∫y(2x)dx` as *solved* — it never checks the
  antiderivative for a leftover `Integrate` (the second-order path does, at 894).
- **A4 (P1, confirmed).** Variation of parameters is silently disabled for
  exponential bases: the Wronskian construction yields `-2e^(x−x)` which
  `.simplify()` does not fold on this path (`differential-equations.ts:876-878`),
  so `y'' − y = eˣ` comes back inert instead of `c₁eˣ + c₂e⁻ˣ + ½xeˣ`.
- **A5 (P1, confirmed).** The parsed-LaTeX entry path is broken for the
  targeted class: `ce.parse("y''(x)+y(x)=0")` canonicalizes the `D` term to
  `Error(incompatible-type)` before `DSolve` sees it (same undeclared-`y`
  typing as A2). All shipped tests feed raw MathJSON through the `lazy`
  operator, so this path has zero coverage.
- **A6 (P2s).** `Apply(Derivative(y), x)` (implicit order 1) never recognized
  (`differential-equation-utils.ts:66` — `op1.op2 === undefined` can never be
  true, `Nothing.N().re` is NaN); absolute 1e-8 tolerances (root-merge vs
  1e-7 conjugate-pairing mismatch can orphan a conjugate → inert);
  `ce.chop` doesn't remove DK noise in exponents; nonhomogeneous support gaps
  (sin/exp forcing, order ≥ 3) — all correctly inert;
  `equationDerivativeCoefficients` recomputed up to 4× per call.
- **Perf/recursion: clean.** `dSolve` is only reachable from the `lazy`
  `DSolve` handler, so its `.simplify()` calls are outside the simplify-rule
  recursion hazard; DK iteration-capped; rk4 deadline-checked. All symbolic
  cases ≤ 60 ms.

## B. Loose-syntax free functions (0.67.0) — silent-meaning-change surface

`src/compute-engine/free-functions.ts:49` switched `toExpression()` to
`ce.parse(input, { strict: false })` for `simplify()/evaluate()/N()/expand()/
factor()/solve()/compile()` string inputs, with **no strict opt-out**.
Perf is clean (single parse, no fallback; measured ≈1.0× vs strict).

- **B1 (HIGH, confirmed).** Valid LaTeX with unbraced multi-digit scripts
  silently changes meaning: `N("x^23")` was `3x²`, now `x²³`; `x2` was `2x`,
  now unbound symbol `x_2`; `a_12+b_12` was `2a₁+2b₁`, now `a_12+b_12`.
  Contradicts the CHANGELOG's "canonical LaTeX input is unaffected" for this
  class. Needs an options pass-through (`simplify(s, { strict: true })`) or a
  CHANGELOG correction — preferably both.
- **B2 (HIGH, confirmed).** `sin^-1 x` parses as `1/sin x` in loose mode while
  `\sin^{-1} x` is `Arcsin(x)` in strict — the same engine gives contradictory
  answers for the standard inverse notation (`parse.ts:2098-2126` lacks the
  `-1 → InverseFunction` case).
- **B3 (HIGH, confirmed).** `atan2(1,2)` → `Arctan(2·(1,2))` — the
  trailing-digit-as-base special case exists only for `log`
  (`parse.ts:2079-2096`); every other name treats the digit as an implicit
  argument.
- **B4 (MEDIUM, confirmed).** Whole-word-only matching backtracks to per-letter
  parsing on a miss, re-injecting the imaginary unit: `2pix` → `2i·p·x`,
  `sin*x` → `i·n·s·x` — the exact mangling class the loose mode was meant to
  fix.
- **B5–B8 (LOW–MED, confirmed).** `sqrt2x` → `√(2x)` (diverges from
  `\sqrt2x` = `√2·x` and AsciiMath convention); `min x` → `Min(x)`;
  `BARE_FUNCTION_MAP` gaps (`acot/asec/acsc`, `nPr` absent → letter soup);
  `x2 → x_2` but `alpha2 → alpha·2`.
- **B9 (new, from the residue triage).** LaTeX-string rule patterns with
  explicit wildcards (`{match: '_a + 1'}`) parse `_a` as `Multiply(_, a)` under
  the lenient parser and never match; MathJSON patterns and `'lhs -> rhs'`
  auto-wildcard rules are unaffected. Needs a doc line or a loud reject.
